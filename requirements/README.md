# Требования

Этот каталог содержит требования проекта, организованные по типам, в соответствии с методологией инженерии требований Карла Вигерса.

## Структура каталога

```
requirements/
├── schemas/                  # Определения JSON-схемы
│   └── requirement.schema.json
├── business/                 # Бизнес-требования (BR-xxx)
├── stakeholder/              # Требования заинтересованных сторон (SR-xxx)
├── functional/               # Функциональные требования (FR-xxx)
├── nonfunctional/            # Нефункциональные требования (NFR-xxx)
├── constraints/              # Ограничения (CR-xxx)
└── interface/                # Требования к интерфейсам (IR-xxx)
```

## Формат ID требований

Каждое требование имеет уникальный ID с префиксом, указывающим его тип:

| Префикс | Тип               | Описание                                           |
|---------|-------------------|------------------------------------------------------|
| BR      | Бизнес            | Высокоуровневые бизнес-цели и задачи                 |
| SR      | Заинтересованные стороны | Потребности и ожидания заинтересованных сторон   |
| FR      | Функциональные    | Конкретное поведение и возможности системы            |
| NFR     | Нефункциональные  | Атрибуты качества (производительность, безопасность)  |
| CR      | Ограничения       | Технические или организационные ограничения           |
| IR      | Интерфейсные      | Требования к взаимодействию с внешними системами      |

## Именование файлов требований

Каждый файл требования должен быть назван строго по ID требования с расширением `.json`:

```
{PREFIX}-{NNN}.json
```

Где `{PREFIX}` — префикс типа требования (`BR`, `SR`, `FR`, `NFR`, `CR`, `IR`), а `{NNN}` — трёхзначный номер.

Примеры: `BR-001.json`, `FR-013.json`, `NFR-002.json`.

## Содержание каталогов

В каталогах с требованиями (`business/`, `stakeholder/`, `functional/`, `nonfunctional/`, `constraints/`, `interface/`) допускаются **только** файлы требований в формате `{PREFIX}-{NNN}.json`. Никакие другие файлы (README, временные файлы, и т.д.) не должны присутствовать.

Каталог `schemas/` содержит только файл схемы `requirement.schema.json`.

## Трассировка

Требования поддерживают двунаправленную трассировку:

- **`traces_from`**: Ссылки на требования более высокого уровня, из которых получено данное требование (обратная трассировка вверх)
- **`traces_to`**: Ссылки на требования более низкого уровня, полученные из данного требования (прямая трассировка вниз)
- **`implementation_files`**: Ссылки на файлы исходного кода, реализующие требование
- **`test_files`**: Ссылки на файлы тестов, проверяющие требование

### Иерархия уровней требований

Типы требований образуют иерархию от высокого уровня к низкому:

```
Уровень 0: BR  (Бизнес-требования)
Уровень 1: SR  (Требования заинтересованных сторон)
Уровень 2: FR, NFR, CR, IR  (Функциональные, нефункциональные, ограничения, интерфейсные)
```

### Ограничения трассировки

1. **Запрет ссылки на себя**: Требование не может ссылаться на себя ни в `traces_from`, ни в `traces_to`.
2. **Запрет циклических ссылок**: Цепочка трассировки не должна содержать циклов (A → B → ... → A).
3. **Направление прямых ссылок (`traces_to`)**: Прямая ссылка допускается только на требования **того же или более низкого уровня** иерархии.
4. **Направление обратных ссылок (`traces_from`)**: Обратная ссылка допускается только на требования **того же или более высокого уровня** иерархии.

### Таблица допустимых прямых ссылок (`traces_to`)

| Источник | Допустимые цели                |
|----------|-------------------------------|
| BR       | BR, SR, FR, NFR, CR, IR      |
| SR       | SR, FR, NFR, CR, IR          |
| FR       | FR, NFR, CR, IR              |
| NFR      | NFR, CR, IR                  |
| CR       | CR, IR                       |
| IR       | IR                           |

## Формат комментариев в исходном коде

Исходные файлы, реализующие требования, должны содержать ссылку в следующем формате:

```cpp
// @req FR-001, FR-002 — Краткое описание того, что реализует этот код
```

Для документации на уровне блоков:

```cpp
/**
 * @req FR-003 — Доступ к JSON-значениям по пути
 * @req FR-004 — Парсинг и сериализация JSON
 */
```

## Валидация

Требования валидируются автоматически в CI двумя уровнями:

```bash
node scripts/validate-requirements.js
node scripts/validate-repo-guard-workflow.js
repo-guard check-diff
```

`scripts/validate-requirements.js` проверяет:
1. Все JSON-файлы соответствуют схеме требований
2. Имена файлов соответствуют паттерну `{PREFIX}-{NNN}.json` и совпадают с ID требования
3. В каталогах с требованиями нет лишних файлов
4. Отсутствие ссылок на себя в трассировке
5. Отсутствие циклических ссылок в трассировке
6. Корректность направления трассировки (прямые ссылки только вниз/на тот же уровень, обратные — только вверх/на тот же уровень)
7. Согласованность двунаправленной трассировки
8. Существование файлов реализации и тестов (только предупреждения)

`repo-guard` с профилем `requirements-strict` проверяет:
1. Все ID внутри JSON-файлов требований разрешаются в существующие requirement IDs
2. Ссылки `@req` в коде, тестах, примерах и скриптах разрешаются
3. Ссылки на требования в Markdown разрешаются
4. Строгие архитектурные заголовки содержат bracketed requirement link
5. Измененные требования и declared PR anchors имеют evidence-файлы
